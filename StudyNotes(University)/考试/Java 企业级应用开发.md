## 二、Spring Boot 核心知识点

### 启动类注解
*【手写笔记：Spring Boot 3特性：自动配置、起步依赖、内嵌服务器】*
1. **@SpringBootApplication**：Spring Boot 应用的启动注解，是一个复合注解，包含 `@Configuration`（标识该类为配置类）、`@ComponentScan`（自动扫描组件）和 `@EnableAutoConfiguration`（启用自动配置）三个核心注解。
   *【手写笔记：如何理解 Spring Boot 自动配置？根据项目中引入的依赖自动配置组件环境 Bean。】*
2. **@EnableScheduling**：用于开启 Spring Boot 应用的定时任务功能，支持 cron 表达式或其他时间策略来定义任务的执行时间。
3. **@SpringBootConfiguration**：与 `@Configuration` 作用相同，但专用于 Spring Boot 应用，明确标识该类为 Spring Boot 的配置类。
4. **@EnableWebMvc**：用于开启 Spring MVC 的配置，允许开发者自定义 MVC 配置而不覆盖 Spring Boot 的自动配置，提供更灵活的 MVC 配置方式。
5. **@EnableTransactionManagement**：开启注解驱动的事务管理，允许在方法上使用 `@Transactional` 注解来声明事务。

### 配置文件
6. **application.yml / application.properties**：Spring Boot 的主要配置文件，支持 YAML 和 Properties 两种格式。YAML 格式更简洁，适合层次化的配置；Properties 格式则更传统，易于理解。
7. **配置优先级**：命令行参数 > 外部配置文件（如 `application-{profile}.yml`） > `bootstrap.yml` > `application.yml / application.properties`。这种优先级设置使得配置更加灵活，可以根据不同环境或需求进行覆盖。
8. **多环境配置**：通过 `application-{profile}.yml` 命名规则来区分不同环境的配置文件，使用 `spring.profiles.active` 属性来激活特定的环境配置。
9. **配置绑定**：使用 `@ConfigurationProperties` 注解可以将配置文件中的属性绑定到 Java 对象上，便于集中管理和使用配置。 *【手写笔记：可读取配置文件的数组类型属性】*
10. **Profile-specific 配置**：在 `application.yml` 文件中可以使用 `---` 分隔符来定义多个 profile 的配置，无需维护多个配置文件，简化了配置管理。

### 内嵌服务器
*【手写笔记：什么是内嵌服务器？集成了 Tomcat 等服务器，无需外部容器，直接运行 jar 包启动的 Web 项目】*
11. **默认服务器**：Spring Boot 默认使用 Tomcat 作为内嵌服务器，无需额外配置即可快速启动 Web 应用。 *【手写笔记：通过 Server.port 设置内嵌服务器端口号】*
12. **支持的内嵌服务器**：除了 Tomcat，Spring Boot 还支持 Jetty 和 Undertow 等内嵌服务器，可以根据项目需求选择合适的服务器。
13. **服务器配置**：通过 `application.yml` 或 `application.properties` 文件可以配置服务器的端口号、上下文路径等参数。
14. **SSL 配置**：支持在 `application.yml` 中配置 SSL 证书，实现 HTTPS 协议，提升应用的安全性。
15. **HTTP/2 支持**：通过简单的配置即可启用 HTTP/2 协议，提升 Web 应用的性能，特别是在高延迟网络环境下效果显著。

### 全局异常处理
*【手写笔记：全局异常处理共有三个注解】*
16. **@RestControllerAdvice / @ControllerAdvice**：用于定义全局异常处理器，结合 `@ExceptionHandler` 注解可以捕获并处理特定类型的异常。
17. **异常处理策略**：可以定义多个 `@ExceptionHandler` 方法来处理不同类型的异常，也可以定义一个通用方法来捕获所有未处理的异常。
18. **自定义异常**：通过定义自定义异常类，可以实现更精细的异常处理逻辑，提升代码的可读性和可维护性。
19. **异常日志记录**：在异常处理方法中记录异常日志，便于问题排查和定位。

### 请求处理
*【手写笔记：由前端传来】*
20. **@RequestBody**：用于将 HTTP 请求体中的 JSON 数据自动绑定到方法参数上，简化数据接收和处理过程。
21. **@RestController**：是 `@Controller` 和 `@ResponseBody` 的组合注解，用于定义 RESTful API 的控制器类，简化代码编写。 *【手写笔记：@Autowired 用于自动注入】*
22. **@RequestMapping**：用于映射 HTTP 请求到控制器方法，支持指定请求路径、方法、参数等，提供灵活的路由配置。
23. **@PathVariable**：用于获取 URL 中的路径变量，便于实现 RESTful 风格的 URL 设计。
24. **@RequestParam**：用于获取请求参数，支持默认值和必填参数等配置。
25. **@RequestHeader**：用于获取请求头信息，便于根据请求头进行业务逻辑处理。
*【手写笔记补充：SpringBoot 测试类注解：@SpringBootTest。 整个 Spring Boot 项目的静态资源放在 resources 目录。】*

### 事务管理
26. **@Transactional**：用于声明方法需要事务管理，支持指定传播行为、隔离级别等属性，确保数据的一致性和完整性。
27. **事务传播行为**：包括 REQUIRED（*【手写笔记：默认的】*）、SUPPORTS、MANDATORY 等七种，用于定义事务的传播方式。
28. **事务隔离级别**：包括 DEFAULT、READ_UNCOMMITTED、READ_COMMITTED 等五种，用于定义事务的隔离级别，防止并发问题。
29. **事务超时**：通过 `@Transactional` 的 `timeout` 属性可以设置事务的超时时间，防止长时间运行的事务占用资源。
30. **事务只读**：通过 `@Transactional` 的 `readOnly` 属性可以将事务设置为只读，提升数据库查询性能。

### 异步任务
*【手写笔记：独立线程执行方法，不阻塞主线程，提升响应速度】*
31. **@Async**：用于标记方法为异步执行（*【手写笔记：标记异步任务】*），需配合 `@EnableAsync`（*【手写笔记：开启异步任务】*）使用，提升应用的响应速度和吞吐量。
32. **线程池配置**：可以通过实现 `AsyncConfigurer` 接口或使用 `@Bean` 注解定义 `ThreadPoolTaskExecutor` 来配置线程池参数。
33. **自定义线程池**：根据业务需求自定义线程池参数，如核心线程数、最大线程数等，优化异步任务执行效率。
34. **异步任务异常处理**：通过实现 `AsyncUncaughtExceptionHandler` 接口可以处理异步任务中的异常，确保异常得到妥善处理。

### 全局响应处理
35. **@RestControllerAdvice / @ResponseBodyAdvice**：用于实现全局响应处理，如统一添加响应头、封装响应体等。
36. **响应体封装**：定义通用响应体类，封装状态码、消息、数据等字段，提升 API 的规范性和易用性。
37. **响应数据过滤**：在全局响应处理器中对返回数据进行过滤或转换，满足前端需求或保护敏感数据。
38. **跨域处理**：通过 `@CrossOrigin` 注解或全局配置解决跨域问题，支持前端跨域访问 API。

### 热部署
*【手写笔记：热部署的底层核心特征行为？spring-boot-devtools！】*
*【手写笔记：在代码中通常为 `<artifactId>spring-boot-devtools</artifactId>`】*
39. **spring-boot-devtools**：通过集成 `spring-boot-devtools` 模块可以实现代码修改后的自动重启应用，提升开发效率。
40. **配置**：启用自动编译功能，并配置 `spring.devtools.restart.enabled=true` 来开启热部署功能。
41. **LiveReload**：配合 LiveReload 插件可以实现浏览器自动刷新，进一步简化开发流程。
42. **排除静态资源**：通过配置可以排除静态资源文件的重启触发，减少不必要的重启操作。
*【手写笔记补充：Spring Boot + redis 连接池 -> lettuce / Jedis (两种)】*

## 三、MyBatis-Plus 核心知识点
*【手写笔记：Spring Boot 整合 Redis 步骤？MyBatis-Plus 核心优势是？】*

### 主键与表映射
*【手写笔记：用法示例 @TableId(type = IdType.AUTO/UUID)】*
43. **@TableId**：用于标记实体类中的主键字段，并指定主键生成策略（*【手写笔记：①自增 ②UUID ③雪花算法 ④手动输入】*）。
44. **@TableName**：用于指定实体类对应的数据库表名，支持自定义表名映射。
45. **@TableLogic**：用于实现逻辑删除功能，通过标记字段实现数据的软删除而非物理删除。
46. **@KeySequence**：用于 Oracle 等使用序列的数据库，指定序列名以实现主键的自增。
47. **@TableField**：用于指定实体类字段与数据库表字段的映射关系，包括字段名、填充策略等。

### CRUD 操作
48. **BaseMapper 接口**：包含基本的 CRUD 方法（*【手写笔记：CRUD核心接口】*），如 `selectList`、`selectOne`、`saveBatch`、`update`、`delete` 等，简化数据库操作。
*【手写笔记：selectCount 查询记录数】*
49. **selectList(null)**：查询所有数据，支持条件查询和分页查询。
50. **selectOne**：查询结果必须唯一，否则会抛出异常。
51. **saveBatch**：批量插入数据，提升插入效率。*【手写笔记：批量操作效率高】*
52. **update**：根据条件更新数据，支持链式调用和动态更新。
53. **delete**：根据条件删除数据，支持链式调用和动态删除。
54. **selectMaps**：查询数据并返回 Map 集合，适用于只需要部分字段的情况，减少数据传输量。

### 条件构造器
*【手写笔记：条件构造器有4个；防止字段写错或编码，避免字符串拼写错误】*
55. **QueryWrapper / LambdaQueryWrapper**：用于构建查询条件，支持链式调用和丰富的条件方法。*【手写笔记：无需手写SQL】*
56. **UpdateWrapper / LambdaUpdateWrapper**：用于构建更新条件，支持链式调用和动态更新。
57. **条件方法**：包括 eq、ne、gt、lt、ge、le、like、between 等，用于构建各种查询条件。
58. **嵌套条件**：通过 and、or 等方法构建嵌套条件，实现复杂的查询逻辑。
59. **函数调用**：在条件构造器中调用数据库函数，如 `apply("date_format({0},'%Y-%m-%d') = {1}", column, dateStr)`，实现灵活的查询。
*【手写笔记补充：MyBatis 的性能插件，分页插件，防全表更新插件(不全，见笔记)，数据权限插件、字段自动填充插件、逻辑删除插件、多租户动态表名插件、数据脱敏插件】*

### 分页插件
*【手写笔记：第11个案例 demo；MyBatis-Plus 要配置分页插件才能进行分页】*
60. **配置**：通过注入 `MybatisPlusInterceptor` 并配置 `PaginationInnerInterceptor`（*【手写笔记：Bean】*）来启用分页功能。
61. **使用**：传入 Page 对象作为参数，返回 Page 对象包含分页信息，如当前页、每页大小、总记录数等。
62. **自定义分页 SQL**：对于复杂查询，可以自定义分页 SQL，并通过 Page 对象传递分页参数，实现灵活的分页查询。
63. **多表分页**：通过关联查询实现多表分页，注意性能优化和避免 N+1 查询问题。

### 乐观锁
*【手写笔记：什么是乐观锁？如何实现乐观锁？ ① @Version 注解 ② 配置乐观锁插件 ③ 设置数据库 version 字段】*
64. **@Version 注解**：用于标记实体类中的版本字段，实现乐观锁机制。*【手写笔记：通过版本号控制并发更新，避免数据覆盖，更新时版本号自增】*
65. **配置**：配置乐观锁插件，执行更新操作时检查版本字段，防止并发更新导致的数据不一致问题。
66. **乐观锁性能优化**：对于高频更新的表，考虑乐观锁对性能的影响，必要时进行优化，如减少版本字段的更新频率。

### 逻辑删除
*【手写笔记：需配置：①实体类字段注解 ②yml配置 ③数据库字段 - 三个都要有】*
67. **默认值**：逻辑删除字段默认值为 0 表示未删除，1 表示已删除，可根据业务需求自定义。
68. **配置**：通过注解配置和 yml 配置指定逻辑删除字段名和删除值、未删除值，实现逻辑删除功能。
69. **逻辑删除与查询**：在查询时自动过滤已逻辑删除的数据，或通过条件构造器包含已逻辑删除的数据，满足不同业务场景的需求。

## 四、Redis 核心知识点
*【手写笔记：Redis 适用场景：①数据缓存 ②分布式锁（防重复提交） ③权限限制 ④消息发布订阅 ⑤学号类型... 等】*
*【手写笔记补充：专门用于操作 String 类型 -> StringRedisTemplate】*

### 数据结构
70. **String**：存储字符串类型数据，支持 SET、GET、INCR、DECR 等操作，适用于计数器、缓存等场景。*【手写笔记：最常用的数据结构】*
71. **List**：适合存储消息队列，支持 LPUSH、RPUSH、LPOP、RPOP 等操作，实现先进先出的数据结构。
72. **Set**：无序不重复数据集合，支持 SADD、SREM、SMEMBERS、SCARD 等操作，适用于去重、集合运算等场景。
73. **ZSet**：有序不重复数据集合，支持 ZADD、ZREM、ZRANGE、ZREVRANGE 等操作，适用于排行榜、范围查询等场景。
74. **Hash**：适合存储键值对形式的对象信息，支持 HSET、HGET、HGETALL、HDEL 等操作，适用于存储对象属性等场景。*【手写笔记：优势：支持字典级操作】*
75. **HyperLogLog**：用于基数统计，占用内存小且计算结果准确度高，适用于大规模数据统计场景。
76. **Geo**：用于存储地理位置信息，支持计算距离、范围查询等操作，适用于 LBS（基于位置的服务）场景。

### 常用命令
*【手写笔记：① Spring Boot 整合 Redis 序列化方式有三种：1. JSON 序列化 2. String 序列化 3. JDK 序列化】*
*【手写笔记：② 整合 Redis 核心操作：RedisTemplate】*
*【手写笔记：③ 整合 Redis 作用：1. 实现缓存 2. 分布式锁 3. 消息队列】*
*【手写笔记：④ 整合 Redis 序列化的目的：数据更容易识别和处理】*
*【手写笔记：PUBLISH Redis 发布消息的命令】*
77. **SET / GET**：存储/获取字符串数据，是最基本的 Redis 命令。
78. **HSET / HGET**：存储/获取 Hash 结构中的字段值，适用于存储对象属性。
79. **LPUSH / RPOP**：List 结构的入队/出队操作，实现消息队列功能。*【手写笔记：还有 LRANGE，共三个操作 LIST】*
80. **INCR / DECR**：计数器操作，对字符串类型的数值进行增减。
81. **EXPIRE**：设置键的过期时间，实现数据的自动过期删除。*【手写笔记：TTL：查看 key 的剩余过期时间】*
82. **DEL**：删除指定键及其对应的值，释放内存空间。
83. **EXISTS**：判断键是否存在，避免对不存在的键进行操作。
84. **FLUSHALL**：清空所有数据库中的数据，慎用此命令。
85. **MSET / MGET**：批量设置/获取多个键的值，提升操作效率。
86. **SCAN**：增量式迭代数据库中的键，避免阻塞主线程，适用于大数据量场景。

### 持久化机制
87. **RDB**：基于快照的持久化方式，定期将内存中的数据写入磁盘文件，实现数据的持久化存储。
88. **AOF**：记录执行命令的持久化方式，将每个写操作命令追加到文件中，实现数据的完整记录。*【手写笔记：记录所有写命令，重启时重新执行命令恢复数据，安全性高】*
89. **混合持久化**：结合 RDB 和 AOF 的优点，既保证数据的完整性又提升恢复速度。
90. **RDB 压缩**：通过配置启用 RDB 文件的压缩，减少存储空间占用，但会增加 CPU 负担。
91. **AOF 重写**：通过配置 AOF 重写策略，减少 AOF 文件体积，提升写入性能。

### 分布式锁
92. **核心命令**：SETNX，用于实现分布式锁的基本命令，确保同一时间只有一个客户端能获取锁。*【手写笔记：利用 Redis 实现互斥锁，解决分布式并发安全问题】*
93. **防止死锁**：为锁设置过期时间，或使用 Redlock 算法确保锁的可靠释放，避免死锁问题。
94. **锁续期**：对于长时间执行的任务，定期续期锁的过期时间，防止任务未完成时锁被释放。
95. **Redisson 客户端**：使用 Redisson 客户端实现更可靠的分布式锁，支持多种锁类型和丰富的功能。
96. **锁等待与超时**：在获取锁时设置等待时间和超时时间，避免无限等待导致资源浪费。

### 序列化
97. **目的**：将数据转换为特定格式进行存储或传输，确保数据的可读性和可恢复性。
98. **推荐方式**：`GenericJackson2JsonRedisSerializer`，支持 JSON 格式的序列化，易于理解和调试。*【手写笔记：4.0 以前的版本】*
99. **其他方式**：`JdkSerializationRedisSerializer`、`StringRedisSerializer` 等，根据业务需求选择合适的序列化方式。
100. **自定义序列化器**：根据业务需求实现自定义的序列化器，提升序列化性能或兼容性。
101. **序列化性能优化**：对于大数据量的存储，考虑序列化性能对 Redis 性能的影响，选择高效的序列化方式。

## 五、Spring Cache 核心知识点
*【手写笔记：Spring 提供的缓存抽象层，整合多种缓存组件】*

### 核心注解
*【手写笔记：通过注解简化缓存操作】*
*【手写笔记标注 102~104 是重点要记的首要词】*
102. **@Cacheable**：用于查询并缓存数据（*【手写笔记：查询操作】*），当方法返回结果时，将结果缓存起来，下次调用时直接从缓存中获取。
103. **@CachePut**：用于更新缓存（*【手写笔记：更新操作】*）且不影响方法执行，每次调用方法时都会更新缓存中的数据。
104. **@CacheEvict**：用于清空缓存（*【手写笔记：清空操作】*），可以在方法执行前或执行后清空指定缓存。
105. **@Caching**：用于组合多个缓存操作，实现复杂的缓存逻辑。
106. **@CacheConfig**：用于统一配置缓存名称等属性，简化缓存配置。
107. **@CacheConfig 的 cacheManager 属性**：指定该类使用的缓存管理器，支持多种缓存实现。
108. **@Cacheable 的 keyGenerator 属性**：指定自定义的 key 生成器，实现复杂的 key 生成逻辑。

### 缓存配置
109. **缓存名称指定属性**：`cacheNames` 或 `value`，用于指定缓存的名称，便于管理和访问。
110. **缓存 key 生成方式**：① 默认生成、② SpEL 表达式自定义、③ 手动指定等，根据业务需求选择合适的 key 生成方式。
111. **缓存管理器**：ConcurrentMapCacheManager、RedisCacheManager 等，支持多种缓存实现和配置。
112. **缓存同步模式**：通过配置实现缓存的同步更新或异步更新，满足不同业务场景的需求。

### 缓存失效原因与防护
113. **缓存穿透防护**：在缓存中设置空值或使用互斥锁防护缓存穿透问题，确保系统的稳定性和性能。
*【手写笔记：缓存失效原因有三个：】*
114. **① 未开启 @EnableCaching**：需要在配置类上添加 `@EnableCaching` 注解来启用缓存功能。
115. **② 序列化失败**：确保缓存的数据类型支持序列化，否则会导致缓存失效或异常。
116. **③ 缓存 key 错误**：确保缓存 key 的唯一性和一致性，避免因 key 错误导致缓存失效。
117. **缓存雪崩**：通过设置不同的过期时间或使用互斥锁避免缓存雪崩问题，确保系统的可用性。
118. **缓存击穿**：对于热点数据，考虑使用永久缓存或互斥锁防护缓存击穿问题，提升系统的响应速度和稳定性。

## 六、定时任务核心知识点
*【手写笔记：什么是 Spring Boot 定时任务？基于注解实现自动执行任务框架，支持 cron 表达式，fixedRate, fixedDelay】*

### 注解
*【手写笔记：开启定时任务：@Scheduled + @EnableScheduling】*
*【手写笔记：固定频率执行 @Scheduled(fixedDelay=5000)：上次执行 5s 后再执行】*
119. **@Scheduled**：用于定义定时任务，支持 cron 表达式、fixedRate、fixedDelay 等属性来定义任务的执行时间和规则。
120. **cron 表达式**：包含 ①秒、②分、③时、④日、⑤月、⑥周等部分，用于定义复杂的任务执行时间规则。*【手写笔记举例："0 0 1 * * ?" 每天凌晨一点一次；"0/5 * * * * ?" 每5秒一次】*
121. **@Scheduled 的 zone 属性**：指定 cron 表达式的时区，确保任务在正确的时间执行。
122. **@Scheduled 的 initialDelay 属性**：设置首次执行任务的延迟时间，避免系统启动时任务立即执行导致的性能问题。

### 实现方式
123. **① 基于 @Scheduled 注解**：直接在方法上添加 `@Scheduled` 注解来定义定时任务，简单易用。
124. **② 结合 TaskScheduler 接口实现**：通过实现 TaskScheduler 接口自定义定时任务执行逻辑，满足更复杂的需求。
125. **③ 使用第三方库如 Quartz**：集成 Quartz 实现更灵活的定时任务管理，支持分布式环境和集群调度与任务持久化。

### 缓存常见问题与解决方案
* **缓存击穿**
  * **问题**：热点 key 过期，大量请求同时访问数据库。
  * **解决方案**：
    1. 互斥锁：缓存过期时，只允许一个线程查询数据库。
    2. 永不过期：热点数据设置永久有效，定时更新。
* **缓存雪崩**
  * **问题**：大量 key 同时过期，所有请求访问数据库。
  * **解决方案**：
    1. 过期时间随机化：避免批量 key 同一时间过期。
    2. 集群高可用：Redis 主从+哨兵，防止单机宕机。
    3. 多级缓存：本地缓存 + Redis 缓存。

### MyBatis-Plus 核心优势
* **核心优势**：MP 无需编写基础 CRUD 的 SQL，通过 `BaseMapper`/`IService` 提供通用方法，支持分页、条件查询、逻辑删除、自动填充等常用功能，大幅减少重复代码。
